- 基準ブランチ: `main`（2026-07-20時点）
- 関連設計: `legalbridge-ui-function-consolidation-design-20260718.md`、`frm-02-app-form-shell.md`

## 1. 結論

全UIを新しい clean / blue 基調へ統一することは可能である。ただし、単純なCSS差替えではなく、次の4点を同時に完了させる必要がある。

1. 画面構造の統一
2. 入力項目・導出項目・閲覧項目・機密項目の整理
3. 書込み経路、データ品質、権限制御の統一
4. 旧画面・旧CSS・重複フォーム・不要APIの物理撤去

本設計では、既存機能を維持しながら、全画面を共通レイアウトと共通コンポーネントへ段階移行する。移行中も正本データ、文書証憑、採番、条件明細、検収・計算処理を壊さないことを最優先とする。

## 4. UI設計原則

### 4.1 一画面一目的

画面は次のいずれかに分類し、異なる目的を混在させない。

| 画面型 | 主目的 |
|---|---|
| Overview | 状況把握、優先順位、件数、アラート |
| List / Search | 対象の検索、絞込み、比較、選択 |
| Detail / Edit | 1エンティティの閲覧・編集 |
| Document Editor | 文書作成、条件確定、証憑生成 |
| Workflow | 検収、計算、支払、統合等の処理実行 |
| Maintenance | 取込、修復、移行、重複統合 |
| Read-only Portal | viewer向け検索・参照 |

### 4.2 書込み口の明確化

- 条件値は文書フォームまたは正規Data Entryからのみ書き込む。
- 一覧・検索・グラフは原則read-onlyとし、編集時は正規フォームへ遷移する。
- 計算結果・スナップショット・採番・監査情報は直接編集させない。
- 旧SSRは検索・閲覧専用とする。

### 4.3 入力項目の状態を明示

すべてのフィールドは次のいずれかを表示する。

- 必須
- 推奨
- 任意
- 自動導出
- 文書から引用
- 確認済
- 要確認
- 閲覧専用

### 4.4 法務データの根拠を見せる

重要項目には、可能な限り次を表示する。

- 元文書
- 元条件明細
- 証憑
- 登録経路
- 最終確認者
- 最終確認日時
- データ品質Issue

## 6. ビジュアルデザイン

### 6.1 基本トーン

- 背景: クールライトグレー
- カード: 白
- 主色: ブルー
- 罫線: クールグレー
- 角丸: 8px基準
- 本文: sans-serif
- コード・文書番号・ID: monospace
- 影: 低コントラスト、カード階層の識別に限定

### 6.2 状態色

| 状態 | トークン |
|---|---|
| 成功・完了 | `success` |
| 注意・確認 | `warning` |
| 情報・引用 | `info` |
| エラー | `destructive` |
| BLOCKER | `severity-blocker` |
| ERROR | `severity-error` |
| WARNING | `severity-warning` |

生の`sky-*`、`emerald-*`、`amber-*`等は段階的に撤去する。

### 6.3 余白

- ページ左右: 24px、狭幅16px
- セクション間: 24px
- カード内: 20px
- フィールド間: 16px
- ラベルと入力: 6px
- アクションバー: 12px 16px

## 11. レスポンシブ

### Desktop 1280px以上

- 本体 + 320px右パネル
- 一覧はテーブル優先
- 作品権利フローは3列

### Tablet 768–1279px

- 右パネルは本文下部またはDrawer
- 一覧は列を削減
- 作品権利フローは1〜2列

### Mobile 767px以下

- 1列
- StickyActionBarは下部固定
- テーブルはカード化または横スクロール
- 破壊操作はメニュー内へ

## 12. アクセシビリティ

- label / htmlFor / aria-describedbyを必須化
- エラーは`role="alert"`
- キーボードのみで全操作可能
- Dialogのfocus trapと復帰
- 色だけで状態を表現しない
- 44px相当のタップ領域
- コントラストWCAG AA
- `prefers-reduced-motion`対応

## 13. 状態設計

全画面で次の状態を実装する。

- 初期ロード
- 再読込
- データなし
- 検索結果なし
- 権限なし
- 部分取得失敗
- 保存中
- 保存成功
- 保存失敗
- 競合
- 読み取り専用
- 廃止ルート

## 15. PR分割方針

- 1PRは1画面型または1モジュールを原則とする。
- デザイン変更とデータ移行を同一PRに混在させない。
- 旧コード削除は代替UIのE2E後に別PRで行う。
- search-apiの機密制御はUI変更とAPI Projection変更を同一PRで完結させる。

## 17. Definition of Done

- 全routeが標準画面型に分類されている。
- 編集可能画面が共通primitiveを使用している。
- 独自`Field()`、独自保存フッター、生の状態色を新設していない。
- 必須・エラー・警告・readonlyが共通表示である。
- 重要項目の根拠文書・証憑へ到達できる。
- viewerに機密項目が返らない。
- 全主要フローのE2Eが成功する。
- Visual Regressionに承認済みbaselineがある。
- 旧SSR編集画面、旧フォーム、旧CSS、不要コードが撤去されている。
- フォーム台帳と項目マトリックスが実装状態と一致している。

## 18. 実行判断

全UIを一度に置換するbig-bang方式は採用しない。LegalBridge AIは文書生成、条件明細、支払・検収、権利情報を扱うため、機能維持を証明できる単位で段階移行する。最終成果物は全画面統一であるが、実装とリリースはWave単位とする。
