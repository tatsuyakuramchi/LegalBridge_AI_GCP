## 2. 「複雑になっている」原因（具体）

1. **§3 が二重レンダリング（最大の肥大）**
   - 現役: `{{#each financial_conditions}}`（~78行, 300-378）。
   - **レガシー fallback**: `金銭条件1_*` / `金銭条件2_*` / `金銭条件3_*` の **同一レイアウトを3回ベタ書き**（~150行, 379-528）。
   - フォームは常に `financial_conditions[]` を送るため、**レガシー fallback はほぼ死蔵**。§3 の約2/3がこの重複。

2. **条件 1/2/3 の意味と既定文がテンプレ直書き（ボードゲーム前提）**
   - 「条件1=自社製造 / 2=ライセンスアウト / 3=プロダクトアウト」と既定タイトル・既定「概要」長文が**両系統に重複してハードコード**（304/317-334, 383/431/481 …）。
   - 汎用の利用許諾（出版・他ジャンル）には合わない。

3. **§1 Grant がボードゲーム専用文言ハードコード**
   - 「ボードゲーム製品」「ゲームルール、テーマ、文面、記号、名称」「パッケージ、カード、ボード、マニュアル」等（197/211/216/223）。区分（BDG/PUB/…）に依らず固定。

4. **SUPPLY_QTY（プロダクトイン）が計算方式表示に未対応【取りこぼし】**
   - §3 の計算方式判定（308）は `BASE_QTY_RATE/BASE_RATE/FIXED/SUBSCRIPTION` のみ。先の product-in 追加で増えた `SUPPLY_QTY`（供給価格×個数）の**ラベルが出ない**（formula_text フォールバック頼み）。

## 3. 簡素化の提案（優先順）

- **T1（最優先・大）: レガシー fallback（金銭条件N_*）を削除**。`financial_conditions[]` 一本化（~150行→0）。残すなら「データ無し時の注記」1行に縮約。古い文書の再描画が要るかだけ確認。
- **T2: 条件の既定タイトル/概要をテンプレ外へ**。`condition_no`→既定文の対応はデータ（フォーム既定値）かヘルパに寄せ、テンプレは `condition_name`/`summary` を素直に出すだけにする。
- **T3: SUPPLY_QTY のラベル追加**（1行）。`{{#if (eq calc_type "SUPPLY_QTY")}}供給価格 × 個数{{/if}}` を計算方式判定に追加。← 小さなバグ修正、すぐ可能。
- **T4: §1 Grant の汎用化**。区分（BDG/PUB/汎用）で文言を出し分け、またはデータ項目化。中規模。

## 4. 進め方の選択肢
- (i) **T3 だけ即修正**（product-in 表示の取りこぼし解消）。
- (ii) **T1+T3** をまず実施（死蔵レガシー削除＋SUPPLY_QTY）でテンプレを大幅スリム化。
- (iii) T1〜T4 を通しで（汎用化まで）。区分別テンプレの設計を伴うので要相談。

> 注: テンプレは disk(`templates/` と `services/worker/templates/`) と DB の二重管理。`sync-templates-to-db.mjs`（TEMPLATE_SOURCE=db）で同期される運用のため、変更は disk 2ファイルを揃え DB 同期も必要。
